## 🐛 Critical Bug Fixes

### Multi-Deck Import — Card Count Now Accurate

**Problem:** When importing multiple card deck files (e.g., Part 1, Part 2, Part 3), the total card count did not reflect the correct number. Instead of 180 cards (60 × 3 decks), only 60 cards were stored.

**Root Cause:** All three scenario deck files used the same internal ID scheme (`SC-001` through `SC-060`). IndexedDB uses the `id` field as the primary key, and `store.put()` performs an upsert — silently overwriting any existing record with the same key. When Part 2 was imported, its `SC-001` overwrote Part 1's `SC-001`, despite being a completely different card. Part 3 then overwrote Part 2.

**Fix:** Four files were updated to implement composite ID generation and two-tier duplicate detection:

| File | Change |
|---|---|
| `parsers.js` | New `generateImportId()` function produces unique composite keys by combining the source filename with the original card ID (e.g., `cissp-scenarios-part-1__SC-001`). Applied to all three parsers: JSON, CSV, and plain-text. Original IDs are preserved in a `sourceId` field for display. |
| `importer.js` | Duplicate detection now uses two checks: (1) exact composite ID match detects same-file re-imports, (2) normalized `term + domain` content key detects true content duplicates across different files. Cards from different files with different content always receive unique IDs and are stored independently. |
| `db.js` | `prepareCardForStorage()` now guarantees a UUID fallback for missing IDs and preserves `sourceId`, `sourceDeck`, and all required array fields (`alsoSee`, `tags`, `relatedCards`). `migrateCards()` updated to fix legacy records missing these fields. |
| `app.js` | Import preview now categorizes duplicates by type (re-import vs. content match). Added null-safety checks. Home view stats refresh automatically after import completes. |

**Result:**

| Scenario | Before (v1.1.1a) | After (v1.1.2) |
|---|---|---|
| Import Part 1 (60 cards) | 60 stored | 60 stored |
| Import Part 2 (60 cards) | 60 stored (Part 1 overwritten) | 120 stored |
| Import Part 3 (60 cards) | 60 stored (Parts 1 & 2 overwritten) | **180 stored** |
| Re-import Part 1 | Silent overwrite, no change | 60 detected as re-import, skipped |

### Extension Popup Crash Fix (popup.js)

**Problem:** `popup.js` threw an uncaught error at line 225 when the popup failed to load data, causing a blank popup with no recovery option.

**Root Cause:** The `catch` block attempted to access `document.getElementById('openNewTabFallback')` immediately after setting `innerHTML` on `popupBody`, without null-checking either element. If `popupBody` was null (DOM not ready or element missing), the entire catch block threw a secondary error.

**Fix:**

| Change | Detail |
|---|---|
| Null-safe `popupThemeToggle` | Added existence check before attaching click listener |
| Null-safe `isDueToday` | Added guard for missing `sm2.nextReviewDate` property |
| Centralized `openStudyDeck()` helper | Wraps `chrome.tabs.create` with API availability check; graceful fallback to `window.open` |
| Null-safe `#popupBody` | Returns early with console error if element is missing |
| Safe streak access | Handles `undefined` streak data with `|| 0` fallback |
| **Catch block rewrite** | Stores element references before use, null-checks both `popupBody` and `openNewTabFallback` before attaching listeners |

## ⚠️ Important: Post-Update Action Required

> **Users with existing imported data must clear their card store and re-import all decks after updating to v1.1.2.**

Previously imported cards have colliding IDs from the old scheme (e.g., multiple cards all stored as `SC-001`). The new composite ID system cannot retroactively fix records that were already overwritten. To get the correct total card count:

1. Open StudyDeck → **Settings** or **Deck Manager**
2. Clear all card data
3. Re-import all deck files (Part 1, Part 2, Part 3, starter deck, etc.)
4. Verify the total count reflects the sum of all imported cards

Study progress (SM2 data, review history) will be reset. If you need to preserve progress, export your data before clearing, then re-import — the export format includes SM2 state which will be restored on re-import.

## 🔧 Technical Details

### Composite ID Format

```
{sanitized-filename}__{original-card-id}

Examples:
  cissp-scenarios-part-1__SC-001
  cissp-scenarios-part-2__SC-001    ← distinct key, no collision
  cissp-scenarios-part-3__SC-001    ← distinct key, no collision
  cissp-starter-deck__DEF-042
```

### Duplicate Detection Matrix

| Incoming Card | Existing Match By | Same Answer? | Action |
|---|---|---|---|
| Same composite ID | ID | Yes | Skip (re-import) |
| Same composite ID | ID | No | Merge (add as alsoSee) |
| Different ID, same term+domain | Content | Yes | Skip (content duplicate) |
| Different ID, same term+domain | Content | No | Merge (add as alsoSee) |
| Different ID, different term | None | — | **Add as new card** |

### Re-Export Compatibility

Cards exported from v1.1.2 will contain composite IDs. When re-imported, the parser detects the `__` separator and preserves the existing composite ID rather than double-prefixing, ensuring clean round-trip export/import cycles.
